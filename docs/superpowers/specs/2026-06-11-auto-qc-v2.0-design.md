# auto-qc v2.0：多规则集 + 逐规则打标

> **Date:** 2026-06-11
> **Branch:** v2.0
> **Status:** Draft

## 1. 背景与目标

### 1.1 动机

当前 v1.x 版本将所有质检规则平铺在一个文件中，一次 LLM 调用同时检查所有规则。这种方式存在两个问题：

1. **规则混淆**：来自不同角度的规则（如人文角度 vs 项目规范）混在一起，LLM 注意力分散，准确度下降
2. **输出不直观**：每条违规占一行，同一个 case 的多个违规拆成多行，不便于逐条查看和统计

### 1.2 目标

1. **多规则集管理**：每个规则集是独立文件，运行时选择 N 个规则集组合使用
2. **逐规则打标**：每条规则独立调 LLM 判断，提升准确度和可溯源能力
3. **宽表输出**：每条对话一行、每个规则一列，最右侧汇总打标详情
4. **统计细化**：规则集维度、规则维度、问题分布多维度统计
5. **安装简化**：不再复制代码到终端目录，skill 目录直跑

## 3. 规则集模型

### 3.1 规则集文件

每个规则集是一个独立的 JSON 文件（skill 中只存结构化数据，不存 Markdown）。

**为什么用 JSON 而非 MD：** 上游（如 auto-pi）产出的 MD 格式可能因版本更新或内容调整而产生格式波动，导致解析失败或信息丢失。JSON 作为稳定中间格式，在写入 skill 时就已完成信息提取和校验，后续运行时不再依赖格式解析，消除解析环节的不确定性。

```json
{
  "name": "auto-pi",
  "display_name": "人文角度",
  "description": "从人文/拟人/真人客服角度评估",
  "rules": [
    {
      "rule_id": "R01",
      "name": "答非所问",
      "severity": "高",
      "description": "AI 的回答与用户的问题无关",
      "detection_logic": "检查 AI 是否回避了用户的核心问题",
      "examples": ["用户: \"这个多少钱?\" AI: \"您可以先了解一下功能\""]
    },
    {
      "rule_id": "R02",
      "name": "过度追问",
      "severity": "中",
      "description": "AI 在用户未回应的情况下反复追问",
      "detection_logic": "检查 AI 是否在同一轮对话中连续追问 3 次以上",
      "examples": []
    }
  ]
}
```

规则 ID 在加载时由框架统一重新编码（如 `auto-pi_R01`、`project_R01`），避免不同规则集之间的 ID 冲突。

### 3.2 规则集注册

框架内置扫描 `rules/` 目录下的 `.json` 文件发现可用规则集：

```python
class RuleSet:
    name: str           # "auto-pi"
    display_name: str   # "人文角度"
    rules: list[Rule]   # 解析后的规则列表，ID 已重编码
```

CLI 参数 `--rule-sets auto-pi,project-standards` 指定要使用的规则集名称（对应 `rules/auto-pi.json` 和 `rules/project-standards.json`）。

## 4. 逐规则打标流程

### 4.1 核心变化

| 维度 | v1.x | v2.0 |
|------|------|------|
| 每次 LLM 调用 | 检查所有规则 | 检查 1 条规则 |
| Prompt 内容 | 全部规则的 JSON 数组 | 仅 1 条规则的描述 |
| 输出结构 | batch_id + rules_checked + results | batch_id + rule_id + results |
| 调用次数 | N 批 × 1 次 | N 批 × M 条规则 |
| 结果合并 | 无（各 batch 直接合并） | 按规则 × 对话交叉合并 |

### 4.2 单规则 Worker Prompt & 校验

```markdown
# 质检任务

你是一名质检员。请检查以下对话是否违反指定的规则。

## 规则

{规则 JSON（仅一条规则）}

## 对话数据

{批次对话 JSON}

## 工作要求

对每一条对话，判断是否违反上述规则。

## 输出格式

{
  "batch_id": 1,
  "rule_id": "R01",
  "results": [
    {
      "id": "001",
      "violates": true,
      "evidence": "用户: XXX | AI: XXX",
      "reasoning": "简要推理过程"
    },
    ...
  ]
}
```

### 4.3 逐条合规校验

每次 LLM 调用返回后，对输出结果做逐条校验——不再是只校验整体结构，而是深入到每条 case 的判断结果。所有校验项失败均触发整批重试：

| 校验项 | 说明 | 处理方式 |
|--------|------|----------|
| JSON 合法性 | 响应可解析为合法 JSON | 整批重试 |
| 结果完整性 | `results` 数组长度 = 批次对话数，ID 全部匹配 | 整批重试 |
| `violates` 有效性 | 每条结果必须有 `violates: true/false` | 整批重试 |
| `evidence` 完整性 | `violates=true` 的必须有非空 evidence | 整批重试 |
| `reasoning` 完整性 | `violates=true` 的必须有非空 reasoning | 整批重试 |

**重试策略：** 整批重试，最多 3 次，指数退避（1s → 2s → 4s）。3 次重试耗尽后仍然校验不通过 → 该规则×该批次标记为失败，这部分对话在该规则下的结果标记为"未知"，继续执行剩余规则。

### 4.4 并发策略

```
对每个规则集 × 每条规则：
  对该规则下的所有批次并发执行 LLM 调用
  
并发数 = 批次数量 × 规则数量（但以信号量限制实际并发）
```

例如：500 条对话、25 条/批 = 20 批、2 个规则集共 13 条规则 → 260 次 LLM 调用同时发出。

### 4.5 结果合并

打标完成后，将各规则的独立结果合并为宽表格式：

```python
# 中间结构
all_results = {
  "R01": {"001": {"violates": True, "evidence": "..."}, ...},
  "R02": {"001": {"violates": False, "evidence": ""}, ...},
  ...
}

# 合并为宽表行
wide_rows = [
  {
    "id": "001",
    "time": "...",
    "intent": "...",
    "rules": {
      "R01": {"result": "违规", "evidence": "...", "rule_name": "答非所问"},
      "R02": {"result": "通过", "evidence": "", "rule_name": "过度追问"},
      ...
    },
    "summary": "答非所问：用户问XX AI答XX\n语气生硬：AI说XX"
  },
  ...
]
```

## 6. 交叉验证适配

逐规则模式下交叉验证也按规则独立进行：

1. 抽样 200 条对话（保持现有 `fixed_sample` 逻辑）
2. 对每条规则分别做复检（1 次 LLM 调用 = 1 条规则 × 抽样对话）
3. 按规则计算 Cohen's Kappa（二分类：违规/通过）
4. Kappa < 0.6 的规则触发第三次裁决（三局两胜）

与 v1.x 的区别：v1.x 的复检是一次调用检查所有规则，v2.0 的复检也是逐规则独立进行。

## 8. 边界情况

### 8.1 规则集为空
某个规则集文件解析出 0 条规则 → 跳过该规则集，在报告中标记为"无规则"，不影响其他规则集运行。

### 8.2 无违规对话
所有对话全部通过 → 宽表中每行全是"通过"，打标详情列为空，统计中违规率 0%。

### 8.3 规则 ID 重编码
不同规则集之间原始 ID 可能重复（如 auto-pi 和 project 都有 R01）→ 加载时统一重编码为 `{规则集名}_{原始ID}`，如 `auto-pi_R01`、`project_R01`。后续所有内部处理和输出列名均使用重编码后的 ID。

### 8.4 LLM 部分失败
某个规则的某批次调用失败 → 只影响该规则×该批次的 25 条对话，其他规则和其他批次不受影响，缺失结果标记为"未知"。
